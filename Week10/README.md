# Smart Pointers. Колекция от указатели. Преговор за първо контролно

Както видяхме [миналата седмица](https://github.com/alexsrebrov/OOP-Practicum-5group/tree/main/Week09), при `Rule of Zero` искаме така да структурираме класовете си, така че употребата на указатели да е минимална (това означава и самата нужда за управление на памет да се сведе до минимум). Това в голяма степен се реализира с някакъв вид wrapper клас, който управлява динамичната памет
* най-често указателите ни управляват някакви символни низове (което се покрива от `std::string`), или колекция от обекти (която можем да заместим със шаблонния клас `std::vector`)
* какво правим обаче, когато класът ни има член-данна, която е указател към друг клас, който обаче не е колекция? Няма смисъл да използваме цял вектор за един указател към един единствен обект. Затова се въвеждат **Smart Pointers**

## Smart Pointers (Умни указатели)
Обвиващ *(wrapper)* клас около обикновен указател, който се грижи и автоматично освобождава паметта.

### 1. unique_ptr
* имаме връзка 1:1 - *1 указател отговаря за 1 обект*
* това означава, че не може да се копира - можем само да го местим с std::move (прехвърляме собственост)
* изпозлва се най-вече, когато имаме композиция на два класа, и единия **има собственост над другия** (като само той има тази собственост)
* в `stl` се създава чрез метода `make_unique`

```c++
class Engine 
{ 
    //...някаква имплементация...
};

class Car 
{
    // искаме да пазим двигателя като указател, а не като чист обект; единствено тази кола има собственост над двигателя
    std::unique_ptr<Engine> engine;
public:
    Car() : engine(std::make_unique<Engine>()) 
    {}
};

```

### 2. shared_ptr
* имаме връзка 1:много - *1 обект бива управляван от много указатели*
* за целта имаме вътрешен брояч, който следи колко указателя сочат дадения обект
* отново автоматично освобождава заделената памет, *но чак когато броячът достигне нула*
* изпозлва се когато няколко класа/обекти **имат споделена собственост над друг обект**
* в `stl` се създава чрез метода `make_shared`

```c++
class Message {
//...някаква имплементация...
};

class User {
public:
    std::shared_ptr<Message> message;
    // или по-практично, std::vector<std::shared_ptr<Message>> messages
};

```
В примера, имаме система като Messenger, където двама потребителя могат да си споделят съобщения. Така и двамата имат собственост над съобщенията, затова използваме `shared_ptr`

### 3. weak_ptr
* сочи към обект, менажиран от shared_ptr
* не увеличава брояча за `shared_ptr`
* ако все пак искаме да използваме `weak_ptr` се въвежда допълнителен брояч за този вид указатели
* не влияе на триенете и може да сочи към вече изтрит обект - по тази причина `weak_ptr` трябва да има проверки дали обектът е изтрит
* в някаква степен може да се каже, че `weak_ptr`, само сочи/"наблюдава" даден обект, без да може да го променя или да има собственост върху него
* метод **lock()** (възможно е да видите и *promote()*) - превръща го в `shared_ptr`

## Колекции
* 1) колекция от обекти
```c++
A* arr = new A[n] // n пъти викаме default конструктор -> т.е. зависим от съществуването му

void add(const A& obj) {
    if (size == capacity) {
        resize();
    }

    arr[size++] = obj // operator= -> копиране
}
``` 

Досега, когато ни се налага да направим resize() на колекция от обекти, трябва да заделим нов, по-голям масив, и да копираме всеки един елемент от стария

![](/Week10/resources/resize.png)

Както може да се досетите, това често копиране се оказва доста скъпа операция. Можем ли да го избегнем?

* 2) колекция от указатели
```c++
A** ptr = new A*[n] { nullptr };

void add(const A& obj) {
    if (size == capacity) {
        resize();
    }

    ptr[size++] = new A(obj) // извиква се копиращ конструктор; до момента на тази позиция имаме nullptr, т.е. елемента/обекта не е инициализиран
}
```

![](/Week10/resources/emtpy.png)

Хубавото на такъв вид колекции е, че не зависим от default конструктор на дадения клас - стойността ни по подразбиране винаги е `nullptr`. Също така, при такива операции не ни е нужно да правим излишно копиране при `resize` или дори при `swap` на два обекта - достатъчно е просто да пренасочим накъде сочат указателите

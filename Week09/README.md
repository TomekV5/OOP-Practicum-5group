# Rule of Zero, STL контейнери и итератори

## STL библиотеката
STL (Standart Template Library) е част от стандартните библиотеки на C++ (и може би най-използваната). В нея имаме различни *структури от данни, както и алгоритми и итератори*, свързани с тях - в детайли ще учите повече в курса по СДА. Засега ви е достатъчно да знаете основните неща

Някои от структурите данни са:
* **std::string**
* **std::vector**
* std::unique_ptr/std::shared_ptr
* std::pair
* std::map
* ...и още много...

Тези, които ще използвате най-често в курса по ООП са **string** (символен низ) и **vector** (най-простата колекция от обекти *с динамичен размер*)

### String
В голяма степен string е просто wrapper клас на char* - т.е. той сам се грижи за динамичната памет на един символен низ

### Vector
Vector е по-интересната от двете структури, тъй като ни представява шаблонна колекция от обекти с динамичен размер, т.е. **можем да пазим произовлен брой от всякакъв вид обекти, като самата логика по работа с динамична памет се поема от vector** (какво точно означава това малко по-надолу)

Всички елементи във вектор се записват един след друг в паметта без допълнителни усложнения, затова той е най-простата колекция в stl

Някои от основните методи на vector:
* *push_back* - добавяме елемент в края на масива
* *pop_back* - премахваме последния елемент от масива
* *insert* - добавяме елемент в подаден от нас индекс
* *erase* - премахваме елемент на подаден от нас индекс
* *clear* - "изчиства" съдържанието на масива

Два по-интерсни метода са ***begin()*** и ***end()*** - *итератори* към началото/края на масива

## Итератори
Вече изяснихме, че имаме различни структури от данни/контейнери (vector, map, queue, ...), всяка от които си има някакъв различен начин на пазене на самите обекти в паметта. Удобно е обаче и да имаме някакъв начин да обходим тези данни - затова се намесват **итераторите**

*Итераторът ни дава начин да обходим дадена колекция, без да ни интерсува как всъщност се пазят данните в паметта*

Обикновено итераторите работят с вътрешен указател/индекс/брояч към текущия елемент и имат следните методи (но може да зависи от конкретната имплементация)
* *hasNext* - казва ни дали сме в края на колекцията или имаме още елементи
* *next / operator++* - дава ни текущия елемент и мести указателя напред

Както можете да предположите, това до голяма степен може да се сведе до wrapper на някакъв вид указател, които просто улеснява работата по местенето му.

Отново, това е за пълнота, по-ясно за ползата от итераторите ще разберете, когато знаете и повече структури от данни в курса по СДА

### Един практичен пример, който можем да ползваме по ООП

*begin() и end()* са такива итератори към началото и края на vector

```c++
// за момента ни е напълно достатъчно итераторът ни да е прост указател
template<typename T>
T* Vector<T>::begin()
{
	return data;
}

template<typename T>
T* Vector<T>::end()
{
	return data + size;
}

template<typename T>
const T* Vector<T>::begin() const
{
	return data;
}

template<typename T>
const T* Vector<T>::end() const
{
	return data + size;
}
```

Ако ви е било обръщано внимание, for цикълът има "втора форма"

```c++
std::vector<int> nums;
...
for (int num : nums) {
    // вместо да достъпваме всеки елемент чрез индекс, който задаваме в цикъла, можем да използваме тази конструкция за да обходим масива елемент по елемент
}
```

Тази конструкция работи с итератори и за да я използваме е нужно да имаме дефинирани begin() и end() - особено важно когато ние сами разписваме колекцията, а не ползваме готова от stl

## Rule of Zero
И вече по същинската част. Разбрахме, че имаме различни видове колекции, както и начини да ги обходим. *За какво са ни нужни обаче?*

Както сигурно сте забелязали, когато пишем Rule of Three/Five имаме почти идентичен код във всеки клас. Въпреки това без тях не можем, тъй като трябва да работим коректно с динамичната памет. В много голяма част от случайте тази динамична памет за символни низове или някакъв вид колекиця. 

Така, за да си спестим този еднообразен код, но и за да работим правилно, използваме **string и vector**, които се грижат за динамичната памет (вътре в тях има разписани Rule of Three/Five, а ние ги ползваме наготово като статични обекти, без да се притесняваме какво се случва с паметта).

Тази концепция да сведем динамичната памет до минимум и да използваме "базови" класове (колекции), които боравят правилно с нея, се нарича **Rule of Zero**
